## Goal
Восстановить работоспособность локального API и чатов после изменения Prisma-схемы, устранив рассинхрон между кодом приложения и текущим состоянием локальной PostgreSQL-базы, из-за которого запрос `GET /chats` падает с ошибкой `The column chats.title does not exist in the current database`.

## Assumptions
- Кодовая база уже ожидает новую схему чатов: `Chat.type`, `Chat.title`, роли участников, групповые чаты и связанные поля.
- Prisma Client уже обновлен или будет обновлен повторно как часть решения.
- Основная проблема сейчас не в фронтенде, а в локальной базе данных, где не применены нужные миграции.
- Проект запускается локально через Docker PostgreSQL на порту `5433`.
- Пользователь хочет сохранить текущие данные, если это возможно, но в случае сильно поврежденной локальной схемы допустим controlled reset локальной dev-базы.

## Plan
### Phase 1: Confirm the schema drift
- Зафиксировать текущий симптом:
  - `GET /chats` возвращает `500`;
  - Prisma кидает `P2022`;
  - отсутствует колонка `chats.title`.
- Подтвердить, что проблема относится именно к базе, а не к запросу:
  - проверить Prisma schema в [schema.prisma](C:\Users\User\Desktop\Project\apps\api\prisma\schema.prisma);
  - проверить, что модель `Chat` действительно содержит `title` и `type`;
  - проверить наличие миграций, которые должны были создать эти поля.
- Сверить локальное состояние:
  - какие миграции есть в репозитории;
  - какие из них реально применены в локальной БД;
  - нет ли частично примененной или сломанной миграции.
- Output / exit criterion:
  - точно известно, какие изменения схемы отсутствуют в базе.

### Phase 2: Align Prisma artifacts with schema
- Повторно привести Prisma-артефакты в актуальное состояние:
  - `prisma generate`;
  - при необходимости убедиться, что generated client соответствует текущей схеме.
- Проверить backend-типизацию после генерации:
  - `pnpm --filter @repo/api typecheck`.
- Убедиться, что после этого остаются только проблемы базы, а не TypeScript/Prisma Client mismatch.
- Output / exit criterion:
  - Prisma Client и код синхронизированы между собой.

### Phase 3: Apply missing database migrations
- Попробовать штатный путь:
  - применить миграции к локальной базе через `prisma migrate deploy`.
- Если запуск migration engine упирается в локальное `EPERM`, пройти один из fallback-путей:
  - запустить команду с повышенным доступом вне ограничения среды;
  - либо выполнить SQL из нужных migration-файлов вручную против локальной dev-базы;
  - либо использовать `prisma migrate dev` в локальной среде, если это стабильнее именно для dev-инстанса.
- После применения миграций проверить, что в таблице `chats` действительно появились:
  - `title`;
  - `type`;
  - и другие новые поля/связанные структуры, если они были добавлены вместе.
- Output / exit criterion:
  - локальная БД соответствует схеме Prisma и перестает падать на `chats.title`.

### Phase 4: Validate dependent chat features
- После выравнивания схемы проверить базовые API-сценарии:
  - `GET /chats`;
  - `POST /chats/direct`;
  - `POST /chats/group`;
  - `GET /chats/:chatId`;
  - group members endpoints.
- Проверить, что роли участников (`creator/admin/member`) читаются и сериализуются без ошибок.
- Убедиться, что список чатов отдается стабильно даже для старых существующих записей в БД.
- Проверить, нет ли новых ошибок на полях:
  - `chat_members.role`;
  - `chats.type`;
  - `chats.title`.
- Output / exit criterion:
  - backend-сценарии чатов работают на актуальной схеме без `500`.

### Phase 5: Recover local data if needed
- Если миграции применились чисто:
  - сохранить текущую локальную базу как рабочую.
- Если локальная база слишком старая или поврежденная:
  - принять controlled reset только для dev-окружения;
  - заново развернуть локальную схему;
  - при необходимости выполнить seed.
- Перед reset явно оценить последствия:
  - потеря локальных тестовых чатов;
  - необходимость заново создать пользователей/данные.
- Предпочтительный порядок fallback:
  - сначала миграции поверх существующей базы;
  - затем ручное исправление схемы;
  - reset только как последний вариант.
- Output / exit criterion:
  - либо база успешно мигрирована без потери данных, либо локальная dev-база заново поднята в рабочем состоянии.

### Phase 6: End-to-end verification
- Перезапустить локальные сервисы после выравнивания схемы:
  - Docker/Postgres;
  - API;
  - Web.
- Проверить пользовательский сценарий целиком:
  - login;
  - открытие `/chat`;
  - загрузка списка чатов;
  - поиск пользователя;
  - создание/открытие direct-чата;
  - открытие placeholder без ошибок.
- Прогнать технические проверки:
  - `pnpm --filter @repo/api typecheck`
  - `pnpm --filter @repo/web typecheck`
  - `pnpm verify:web`
- При наличии времени или необходимости:
  - локальный smoke-test сценария чатов.
- Output / exit criterion:
  - dev-окружение снова стабильно, а ошибка `GET /chats 500` исчезла.

## Risks and Dependencies
- Риск: migration engine снова упрется в `EPERM` и не сможет подключиться к базе стандартным способом.
  - Mitigation: использовать повышенные права или выполнить SQL миграции вручную для локальной dev-базы.
- Риск: локальная БД отстала не на одну колонку, а на целый набор миграций.
  - Mitigation: проверять не только `chats.title`, а весь набор новых структур схемы чатов.
- Риск: старые данные в локальной базе не соответствуют новым ограничениям схемы.
  - Mitigation: сначала пробовать миграцию с сохранением данных, reset использовать только как крайний fallback.
- Риск: после исправления базы всплывут вторичные ошибки в group chat или membership flows.
  - Mitigation: отдельно пройти ручки групп и ролей после восстановления `GET /chats`.
- Dependency: решение зависит от реального состояния Docker PostgreSQL и возможности применить локальные Prisma migrations.
