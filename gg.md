## Goal
Устранить блокировку локального dev-запуска из-за занятого порта `4000`, стабилизировать запуск `pnpm dev` и исключить повторение сценария со “старым” API-процессом, зависшими PID и ложным ощущением, что backend снова сломан.

## Assumptions
- Текущая проблема уже сместилась с `ERR_CONNECTION_REFUSED` на конфликт локального dev-процесса на `4000`.
- Фронтенд и backend в проекте должны запускаться через единый `pnpm dev`.
- Проверка перед запуском идет через [run-dev-check.mjs](/C:/Users/User/Desktop/Проект/scripts/run-dev-check.mjs) и [dev-port-utils.mjs](/C:/Users/User/Desktop/Проект/scripts/dev-port-utils.mjs).
- База на `5433` уже поднимается через Docker и является обязательной зависимостью для API.
- Ошибка `Cannot find module 'ioredis'` уже была следствием устаревшего/грязного локального состояния и не должна считаться основной корневой причиной, если зависимости уже установлены.

## Plan
### Phase 1: Диагностировать, кто именно занимает `4000`
- Проверить текущее состояние порта `4000`.
- Подтвердить:
  - это действительно живой `node.exe`,
  - это процесс нашего backend,
  - или это зависший старый `pnpm --filter @repo/api start/dev`.
- Если PID есть, определить, не остался ли он после фонового запуска или аварийного завершения `pnpm dev`.
- Проверить, не находится ли порт в переходном TCP-состоянии сразу после `taskkill`, когда preflight еще воспринимает его как занятый.
- Output / exit criterion:
  - есть точный ответ, почему preflight считает `4000` занятым.

### Phase 2: Привести локальное окружение к чистому состоянию
- Завершить все старые backend-процессы, связанные с `@repo/api`.
- Подтвердить, что:
  - `3000` свободен,
  - `4000` свободен,
  - `5433` слушается PostgreSQL-контейнером.
- Проверить, нет ли фоновых `node`/`powershell` процессов, которые повторно поднимают API без участия текущего терминала.
- Проверить, что после очистки окружения `pnpm dev:check` проходит без блокировок.
- Output / exit criterion:
  - preflight полностью зеленый и разрешает запуск dev-среды.

### Phase 3: Перепроверить backend-зависимости и запуск API
- Подтвердить, что в локальном workspace доступны:
  - `ioredis`
  - `@socket.io/redis-adapter`
- Проверить, что `@repo/api`:
  - собирается,
  - стартует в `dev`,
  - стартует в `start`,
  - не падает сразу после старта.
- Проверить, что startup лог доходит до:
  - Nest routes,
  - `/health`,
  - `Nest application successfully started`.
- Проверить, что `EADDRINUSE` появляется только при реальном конфликте порта, а не из-за двойного запуска внутри скриптов.
- Output / exit criterion:
  - API можно стабильно поднять в одиночном экземпляре на `4000`.

### Phase 4: Проверить orchestration сценарий `pnpm dev`
- Проверить, как именно запускаются `web` и `api` внутри `pnpm dev`.
- Подтвердить последовательность:
  - preflight,
  - старт web,
  - старт api.
- Проверить, не остается ли зависших подпроцессов после:
  - `Ctrl+C`,
  - падения `api`,
  - падения `web`,
  - аварийного завершения окна.
- Если нужно, проверить `scripts/run-dev-start.mjs`, чтобы понять:
  - корректно ли он завершает дочерние процессы,
  - нет ли сценария, где `api` остается жить после прерванного `pnpm dev`.
- Output / exit criterion:
  - `pnpm dev` стартует и останавливается предсказуемо, без “сиротских” backend-процессов.

### Phase 5: Улучшить preflight-проверку `4000`
- Перепроверить логику [dev-port-utils.mjs](/C:/Users/User/Desktop/Проект/scripts/dev-port-utils.mjs):
  - как определяется, что порт занят,
  - как читается PID,
  - почему может появляться `PID unavailable`.
- Проверить сценарий, когда:
  - порт уже освобожден,
  - но проверка еще видит временное состояние,
  - и ошибочно блокирует старт.
- При необходимости улучшить проверку:
  - различать `LISTENING` и переходные состояния,
  - повторять короткую проверку перед фейлом,
  - показывать более точное сообщение, если процесс уже убит.
- Output / exit criterion:
  - preflight не дает ложных блокировок после нормальной очистки порта.

### Phase 6: Защитить запуск от повторения проблемы
- Добавить или уточнить dev-памятку:
  - как правильно завершать `pnpm dev`,
  - что делать при занятом `4000`,
  - когда использовать `taskkill /PID ... /F`.
- При необходимости добавить безопасный helper-сценарий:
  - очистка зависшего backend-процесса,
  - повторный preflight,
  - запуск dev-среды.
- Если окажется, что orphan-processes случаются часто, рассмотреть доработку orchestration-скрипта для более агрессивного cleanup дочерних процессов.
- Output / exit criterion:
  - следующий локальный запуск не требует ручного расследования, если старый API снова останется в памяти.

## Risks and Dependencies
- Риск: порт `4000` занимает не наш API, а другой локальный сервис.
  - Mitigation: всегда подтверждать конкретный PID и тип процесса перед завершением.
- Риск: `pnpm dev` оставляет сиротские процессы после `Ctrl+C`.
  - Mitigation: проверить orchestration-скрипт и поведение дочерних процессов на Windows.
- Риск: preflight слишком агрессивно трактует временное TCP-состояние как полноценную блокировку.
  - Mitigation: доработать проверку и добавить короткий повторный polling перед ошибкой.
- Риск: разработчик снова увидит старое сообщение про `ioredis` и примет его за текущую основную проблему.
  - Mitigation: разделять “ошибки старого запуска” и “текущее состояние после reinstall/cleanup”.
- Dependency: доступ к локальным Windows process/netstat данным.
- Dependency: Docker PostgreSQL на `5433`.
- Dependency: установленный актуальный `node_modules` после `pnpm install`.
