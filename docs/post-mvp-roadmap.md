# Post-MVP Roadmap

## Goal
Зафиксировать следующий продуктовый этап после завершения MVP и выбрать первый релизный батч, который даст заметную ценность пользователю без чрезмерного роста сложности.

## Assumptions
- Текущий MVP уже закрывает базовый direct chat flow: auth, direct chats, history, realtime, unread/read status.
- В ближайшей итерации важно усилить пользовательскую ценность, а не перестраивать архитектуру целиком.
- Production-модель на первом этапе остается `VPS + Docker Compose + Caddy`.
- Работы должны укладываться в один контролируемый релизный батч, а не в большой многофункциональный переписывательский цикл.

## Recommended Order
### Batch 1: File attachments
Почему первым:
- это самое заметное функциональное улучшение для реального использования чата
- хорошо ложится на текущую модель direct chats и messages
- не требует немедленного усложнения membership logic, ролей и fan-out
- можно выпустить инкрементально: сначала один файл на сообщение, потом расширять

Цель батча:
- пользователь может прикрепить файл к сообщению и отправить его в direct chat
- собеседник видит вложение в истории и может открыть или скачать его

Выходной критерий:
- вложения работают локально и в production-сценарии
- добавлены API/UI/e2e проверки

### Batch 2: Search in current chats
Почему вторым:
- после появления более богатой истории сообщений поиск начинает приносить больше пользы
- технически это заметно проще, чем групповые чаты
- можно сначала сделать поиск по текущему чату, а уже потом думать о глобальном поиске

Цель батча:
- пользователь может искать сообщения в выбранном чате по тексту
- результаты ведут к нужному сообщению в истории

Выходной критерий:
- есть минимальный search UX, API endpoint и понятное поведение на пустых результатах

### Batch 3: Group chats
Почему позже:
- это самый дорогой по сложности скачок после MVP
- потребует расширения модели чатов, ролей, создания/редактирования участников, unread logic и UI shell
- после появления групп резко вырастут требования к realtime fan-out и к тестовому покрытию

Цель батча:
- пользователь может создать группу, добавить участников и вести групповую переписку

Выходной критерий:
- группы работают как отдельный устойчивый релизный слой, а не как «быстрый хак поверх direct chats»

## Deferred Until Later
Эти задачи пока не стоит брать в первый post-MVP релиз:
- push-уведомления
- Redis adapter
- редактирование и удаление сообщений
- WebRTC-звонки
- object storage abstraction уровня S3

Причина:
- они полезны, но либо инфраструктурно тяжелее, либо зависят от того, как стабилизируются вложения, поиск и группы

## Why This Sequence
- `Attachments first` усиливает core messaging без смены фундаментальной модели чатов
- `Search second` повышает удобство работы с выросшей историей
- `Groups third` идет после того, как direct chat experience станет богаче и стабильнее
- Такой порядок снижает риск распыления: каждая следующая задача становится лучше определена после предыдущей

## Immediate Recommendation
Первый post-MVP релизный батч: **file attachments**.

Для этого батча нужно:
1. Зафиксировать storage strategy для MVP.
2. Описать schema/API/UI изменения.
3. Реализовать backend upload flow.
4. Реализовать UI выбора и отображения вложения.
5. Добавить тесты и обновить production runbook.
