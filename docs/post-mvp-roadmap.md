# Post-MVP Roadmap

## Goal
Зафиксировать следующий продуктовый этап после завершения MVP и держать post-MVP развитие как последовательность отдельных релизных батчей, а не один распухший backlog.

## Current Status
- `Batch 1: File attachments` реализован
- Следующий рекомендуемый батч: `Search in current chats`
- Затем: `Group chats`

## Recommended Order
### Batch 1: File attachments
Статус:
- Completed

Что вошло:
- один файл на сообщение
- direct chats only
- attachment-only сообщения без текста
- inline preview для изображений
- карточки файлов для PDF/TXT и других поддержанных типов
- защищенный download endpoint с проверкой membership
- upload volume в production compose
- API e2e coverage для upload/download

Итог:
- вложения работают локально и включены в текущую кодовую базу

### Batch 2: Search in current chats
Почему следующим:
- после появления вложений и более богатой истории сообщений поиск начинает приносить больше пользы
- технически это заметно проще, чем групповые чаты
- можно сначала сделать поиск по текущему чату, а уже потом думать о глобальном поиске

Цель батча:
- пользователь может искать сообщения в выбранном чате по тексту
- результаты ведут к нужному сообщению в истории

Выходной критерий:
- есть минимальный search UX
- есть API endpoint поиска по текущему чату
- есть понятное поведение на пустых результатах

### Batch 3: Group chats
Почему позже:
- это самый дорогой по сложности скачок после MVP
- потребует расширения модели чатов, ролей, создания/редактирования участников, unread logic и UI shell
- после появления групп резко вырастут требования к realtime fan-out и тестовому покрытию

Цель батча:
- пользователь может создать группу, добавить участников и вести групповую переписку

Выходной критерий:
- группы работают как отдельный устойчивый релизный слой, а не как быстрый хак поверх direct chats

## Deferred Until Later
Эти задачи пока не стоит брать в ближайший релизный батч:
- push-уведомления
- Redis adapter
- редактирование и удаление сообщений
- WebRTC-звонки
- object storage abstraction уровня S3

Причина:
- они полезны, но либо инфраструктурно тяжелее, либо зависят от того, как стабилизируются поиск и группы

## Why This Sequence
- `Attachments first` уже усилил core messaging без смены фундаментальной модели чатов
- `Search second` повышает удобство работы с выросшей историей
- `Groups third` идет после того, как direct chat experience стал богаче и стабильнее
- Такой порядок снижает риск распыления: каждая следующая задача становится лучше определена после предыдущей

## Immediate Recommendation
Следующий post-MVP релизный батч: **search in current chats**.

Для этого батча нужно:
1. Зафиксировать search UX в текущем чате.
2. Описать schema/API/UI изменения.
3. Реализовать backend search endpoint.
4. Реализовать frontend results panel и jump-to-message flow.
5. Добавить тесты и обновить status-файл.
