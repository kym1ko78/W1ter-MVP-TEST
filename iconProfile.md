## Objective
Перестроить левую боковую панель чатов в стиле примера: оставить слева только список чатов и единый поиск, а профиль, создание группы и вспомогательные действия перенести в выезжающее меню по кнопке с иконкой из трех полосок, не меняя текущую цветовую схему, стилистику и общий визуальный язык приложения.

## Constraints
- Сохранить текущие цвета, бордеры, скругления и общую светлую стилистику интерфейса.
- Не ломать существующие сценарии: открытие профиля, logout, поиск пользователей, создание direct-чата, создание группы, глобальный поиск.
- Не ухудшить адаптивность: решение должно работать и на desktop, и на узких экранах.
- Не дублировать поиск: в левой панели должен остаться один объединенный поиск вместо нескольких разрозненных блоков.
- Новое меню по кнопке должно быть доступным с клавиатуры и логично закрываться по клику вне панели или по `Esc`.

## Architecture / Approach
- Основной рефакторинг сосредоточить в [chat-shell.tsx](C:\Users\User\Desktop\Project\apps\web\components\chat-shell.tsx), так как текущий `aside` уже содержит все нужные блоки.
- Пересобрать левый `aside` в два слоя:
  - компактная постоянная панель со списком чатов и единым поиском;
  - скрываемая вспомогательная панель или drawer-меню с профилем, logout, созданием группы и связанными действиями.
- Состояние открытия меню вынести в локальный state компонента `ChatShell`.
- Существующую бизнес-логику не переписывать глубоко, а переиспользовать уже готовые state и mutations, меняя в первую очередь layout и точки входа.

## Plan
### Phase 1: Audit and target structure
- Зафиксировать текущие блоки inside `chat-sidebar`: профиль, logout, поиск пользователей, блок новой группы, global search, список чатов.
- Разделить их на две зоны ответственности:
  - постоянная зона: единый поиск + список чатов;
  - вторичная зона: профиль, logout, создание группы, дополнительные действия.
- Определить, какие из текущих полей можно объединить в один поисковый сценарий без потери функций.
- Выбрать форму меню:
  - desktop: левый drawer/overlay, открываемый по кнопке с тремя полосками;
  - mobile: тот же механизм, но с приоритетом на full-height overlay.
- Output / exit criterion:
  - есть точная карта, что остается в aside постоянно, а что уезжает в меню.

### Phase 2: Sidebar simplification
- Перестроить верхнюю часть левой панели по мотивам примера:
  - добавить кнопку с иконкой меню слева;
  - рядом разместить один объединенный поиск.
- Удалить из постоянного aside крупный профильный блок, кнопку `Exit`, отдельный `Найти пользователя` и отдельный `Global search`.
- Собрать единый поисковый input с нейтральным плейсхолдером вроде `Поиск` или `Чаты, пользователи, сообщения`.
- На уровне логики определить поведение объединенного поиска:
  - использовать текущее `globalSearch` как базовый источник для смешанного поиска;
  - при необходимости встроить быстрый direct-start по найденным пользователям в общий список результатов;
  - решить, нужны ли отдельные секции результатов `Чаты / Пользователи / Сообщения` прямо под единым полем.
- Сохранить ниже единый scrollable список чатов как основной контент панели.
- Output / exit criterion:
  - aside визуально стал компактным и похожим по структуре на референс: верхняя строка управления + список чатов.

### Phase 3: Drawer menu with profile actions
- Создать выезжающее меню, открываемое по кнопке с тремя полосками.
- Перенести внутрь меню:
  - карточку текущего пользователя;
  - ссылку на профиль;
  - logout;
  - блок создания группы;
  - при необходимости дополнительные действия-заглушки, если нужен каркас под будущие разделы.
- Стилистически повторить текущий язык интерфейса:
  - те же цвета;
  - мягкие бордеры;
  - белые или светлые surface-блоки на существующем фоне;
  - без попытки копировать Telegram один в один, только композиционный паттерн.
- Добавить overlay и закрытие по клику вне drawer.
- Добавить закрытие по `Esc`.
- Добавить корректный `focus`-сценарий:
  - фокус на кнопке открытия;
  - при открытии фокус уходит в drawer;
  - при закрытии возвращается на trigger.
- Output / exit criterion:
  - вспомогательные действия доступны через отдельное меню и не перегружают постоянную левую панель.

### Phase 4: Group creation flow migration
- Перенести весь существующий UI создания группы в drawer без потери логики:
  - `groupTitleDraft`;
  - `groupSearch`;
  - выбранные участники;
  - поиск участников;
  - `createGroupChatMutation`;
  - показ ошибок `groupCreateError`.
- Подправить layout блока создания группы под формат вертикального drawer:
  - компактный заголовок;
  - поля с адекватными отступами;
  - список выбранных участников;
  - результаты поиска участников;
  - нижняя CTA-кнопка.
- Проверить, что создание пустой группы и группы с участниками работает так же, как раньше.
- Output / exit criterion:
  - создание группы полностью доступно из drawer и не занимает место в основном aside.

### Phase 5: Unified search behavior
- Определить точное UX-правило для единого поиска:
  - чаты;
  - пользователи;
  - сообщения.
- Пересобрать рендер результатов так, чтобы пользователь видел все в одном месте, но с понятным разделением секций.
- При необходимости убрать отдельный `user-search-results` и свести его к общему `global-search-results`.
- Добавить понятные переходы:
  - к найденному чату;
  - к direct-чату через создание или открытие;
  - к сообщению с навигацией по `chatId/messageId`.
- Убедиться, что пустые, loading и error-state остаются информативными.
- Output / exit criterion:
  - слева реально остался один поиск, покрывающий людей, чаты и сообщения.

### Phase 6: Responsive polish and interaction states
- Проверить desktop-композицию:
  - верхняя строка с меню и поиском;
  - список чатов занимает максимум полезной высоты;
  - drawer не конфликтует с основной областью переписки.
- Проверить tablet/mobile:
  - кнопка меню доступна;
  - drawer открывается поверх интерфейса;
  - поля и кнопки не ломают сетку.
- Проверить состояния:
  - пустой список чатов;
  - длинные имена;
  - длинные email;
  - длинные результаты поиска;
  - открытое меню + активный чат.
- Убедиться, что hover/focus/active-состояния согласованы с текущим дизайном.
- Output / exit criterion:
  - новый layout стабилен на всех ключевых ширинах и не ломает UX.

## Risks and Dependencies
- Риск: объединение поисков может сломать привычный direct-search сценарий.
  - Mitigation: оставить внутри общего результата явный блок `Пользователи` с тем же действием создания direct-чата.
- Риск: drawer начнет конфликтовать с текущей сеткой `chat-shell`.
  - Mitigation: строить меню как overlay внутри `chat-shell`, а не как часть grid-колонки.
- Риск: перенос group UI в меню ухудшит читаемость на мобильных экранах.
  - Mitigation: сделать drawer вертикальным, scrollable и с компактными секциями.
- Риск: много интерактивных состояний в одном компоненте усложнят поддержку.
  - Mitigation: при необходимости выделить `SidebarMenuDrawer` и `UnifiedSidebarSearch` в отдельные локальные компоненты.
- Dependency: текущая логика поиска и создания групп уже реализована в `chat-shell.tsx`; план опирается на ее переиспользование, а не на переписывание backend/API.

## Sequencing Notes
- Сначала нужно упростить layout и определить постоянную структуру aside.
- Потом переносить действия в drawer, иначе интерфейс временно станет перегруженным или частично недоступным.
- Объединение поиска лучше делать после стабилизации layout, чтобы не смешать визуальный и логический рефакторинг в один неуправляемый шаг.
