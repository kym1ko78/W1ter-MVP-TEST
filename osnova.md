## Goal
Собрать единый план следующих работ после завершения MVP: сделать удобный локальный `pnpm dev`, подготовить реальный production deploy и определить следующий продуктовый этап после MVP.

## Assumptions
- Текущий MVP уже стабилен: локальный запуск, `smoke`, API e2e и UI e2e работают.
- Основной стек остается прежним: `Next.js + NestJS + Socket.IO + PostgreSQL + Prisma`.
- Среда разработки пользователя остается Windows, поэтому DX и тестовый запуск должны учитывать занятые порты и особенности `pnpm`/`Next.js`.
- Production deploy нужен как следующий практический шаг, но без немедленного перехода на сложную инфраструктуру вроде Kubernetes.
- Post-MVP функции будут делаться поэтапно, а не все сразу.

## Plan
### Workstream 1: Сделать удобный и предсказуемый `pnpm dev`
#### Phase 1: Диагностика локального запуска
- Зафиксировать текущие точки отказа локального запуска: занятые `3000`, `4000`, `5433`, stale процессы и конфликт `pnpm dev` с отдельными e2e-сценариями.
- Описать, какие порты считаются основными, а какие резервными.
- Проверить, какие скрипты уже есть для ручной проверки готовности `api` и `web`, и что можно переиспользовать.
- Output / exit criterion: есть список реальных локальных сценариев падения и желаемое поведение для каждого.

#### Phase 2: Добавить preflight-проверку перед `dev`
- Создать отдельный `dev-check` или `dev-start` helper-скрипт.
- Скрипт должен проверять, заняты ли `3000` и `4000`.
- Если порт занят, скрипт должен показывать понятное сообщение: какой порт занят, каким PID, и что с этим делать.
- Если это допустимо, можно автоматически предлагать fallback-порты для `web`, но не менять `api` молча без явного логирования.
- Добавить безопасный режим: либо завершить запуск с понятной ошибкой, либо стартовать в fallback-конфигурации.
- Output / exit criterion: пользователь получает понятный сценарий запуска без ручного угадывания причин падения.

#### Phase 3: Упорядочить локальные команды
- Разделить сценарии на:
- `pnpm dev` — обычная разработка
- `pnpm dev:web` и `pnpm dev:api` — точечный запуск
- `pnpm test:ui:e2e:auto` — полностью изолированный e2e
- Убедиться, что документация прямо запрещает опасные сочетания вроде одновременного общего `pnpm dev` и сценариев, которые пишут в ту же build-папку.
- Обновить `README.md` и status-файлы кратким блоком «что можно запускать одновременно, а что нельзя».
- Output / exit criterion: локальный DX формализован и больше не зависит от памяти пользователя.

#### Phase 4: Минимальная автоматизация восстановления
- При необходимости добавить helper для показа или завершения stale процессов на нужных портах.
- Рассмотреть безопасный сценарий без авто-`taskkill`: сначала подсказка, потом отдельная команда на очистку портов.
- Добавить короткий сценарий для запуска «после перезагрузки ПК».
- Output / exit criterion: запуск локальной среды укладывается в понятный короткий ритуал.

### Workstream 2: Реальный production deploy
#### Phase 1: Выбрать целевую production-модель
- Зафиксировать, что для первой production-версии используется `VPS + Docker Compose + reverse proxy + domain`.
- Определить минимальный набор окружения:
- 1 VPS
- Docker Engine + Compose plugin
- PostgreSQL в Docker или managed Postgres
- reverse proxy с HTTPS
- Decide point: использовать встроенный контейнер Postgres или вынести БД отдельно.
- Output / exit criterion: выбран конкретный production topology для первой выкладки.

#### Phase 2: Подготовить production-конфигурацию
- Проверить и допилить `.env.production.example`.
- Подготовить итоговый список production-переменных:
- `DATABASE_URL`
- `JWT_ACCESS_SECRET`
- `JWT_REFRESH_SECRET`
- `API_PORT`
- `API_CORS_ORIGIN`
- `NEXT_PUBLIC_API_URL`
- `NEXT_PUBLIC_SOCKET_URL`
- Проверить `docker-compose.production.yml`, `Dockerfile.web`, `Dockerfile.api` на реальную совместимость между собой.
- Зафиксировать production-порты и схему доменов, например `app.domain` и `api.domain`.
- Output / exit criterion: production-конфиг можно заполнить без догадок.

#### Phase 3: Подготовить инфраструктурный runbook
- Составить пошаговый deploy-runbook:
- подключение к серверу
- установка Docker
- клонирование репозитория
- создание `.env.production`
- запуск контейнеров
- применение миграций Prisma
- первичная проверка `health`
- Добавить сценарий rollback: как откатиться к предыдущему контейнеру или коммиту.
- Добавить список post-deploy проверок: логин, регистрация, чат, отправка сообщения, websocket.
- Output / exit criterion: есть воспроизводимый документированный деплой без импровизации.

#### Phase 4: Первый реальный прогон deploy
- Развернуть проект на сервере в staging или production-like окружении.
- Проверить:
- доступность `web`
- доступность `api`
- миграции
- cookies/refresh flow
- websocket realtime
- CORS
- Исправить различия между локальной и production-средой, если они проявятся.
- Output / exit criterion: проект реально доступен по внешней ссылке.

#### Phase 5: Базовая эксплуатация
- Добавить простой operational checklist:
- как смотреть логи
- как перезапускать сервисы
- как обновлять контейнеры
- как делать backup базы
- как проверять здоровье приложения
- Зафиксировать минимальный мониторинг: health endpoint, доступность сайта, размер логов.
- Output / exit criterion: проект можно не только задеплоить, но и поддерживать.

### Workstream 3: Следующий этап после MVP
#### Phase 1: Приоритизация post-MVP backlog
- Разделить backlog на три уровня:
- ближайшие улучшения UX
- функциональные расширения чатов
- инфраструктурное усиление
- Сформировать short-list первой следующей итерации из 2-3 крупных задач, а не всей очереди сразу.
- Рекомендуемый первый shortlist:
- вложения файлов/изображений
- групповые чаты
- поиск по сообщениям
- Output / exit criterion: есть согласованный следующий релизный скоуп.

#### Phase 2: Спроектировать каждую крупную функцию до кода
- Для вложений:
- выбрать storage strategy
- описать ограничения по размеру и типам файлов
- определить модели БД и API
- Для групповых чатов:
- продумать модели `chat_members`, роли, создание группы, редактирование участников
- Для поиска по сообщениям:
- определить минимальный UX и тип поиска: по текущему чату или глобально
- Output / exit criterion: каждая следующая фича имеет схему данных, API и UI-поведение до реализации.

#### Phase 3: Реализация следующего релизного батча
- Идти итерациями по одной теме:
- сначала schema/API
- потом frontend
- потом realtime или производственные детали
- потом тесты
- Не смешивать в один батч файлы, группы, поиск, push и Redis без необходимости.
- После каждой функции обновлять smoke/e2e тесты и status-файл.
- Output / exit criterion: следующий релиз строится контролируемо, без раздувания объема.

#### Phase 4: Усиление платформы после новых функций
- Когда появятся группы и больший fan-out, оценить необходимость Redis adapter.
- Когда появятся файлы, оценить storage lifecycle, очистку и лимиты.
- Когда появится production-трафик, расширить логирование и добавить базовые метрики.
- Переходить к push-уведомлениям и WebRTC только после стабилизации core messaging функций.
- Output / exit criterion: рост сложности идет по потребности, а не заранее.

## Dependencies and Risks
- Риск: локальный DX останется хрупким на Windows из-за портов и stale процессов.
- Mitigation: preflight-check, явные сообщения об ошибках, отдельные helper-скрипты.

- Риск: production deploy застрянет не на коде, а на DNS, HTTPS, reverse proxy и env-настройках.
- Mitigation: сначала staging-like прогон, потом основной домен.

- Риск: backlog после MVP разрастется быстрее, чем команда сможет его стабильно выпускать.
- Mitigation: брать только 1 релизный батч за раз и завершать его тестами и документацией.

- Риск: добавление файлов и групп резко увеличит сложность модели чатов и UX.
- Mitigation: сначала отдельный дизайн-док по каждой функции, затем реализация малыми шагами.

- Риск: push, Redis и WebRTC будут добавлены слишком рано и размоют фокус.
- Mitigation: считать их задачами следующего уровня зрелости, а не ближайшего релиза.

## Suggested Sequencing
1. Сначала закрыть Workstream 1, потому что удобный `dev`-цикл ускорит все остальные задачи.
2. Затем пройти Workstream 2 и сделать реальный deploy, чтобы увидеть проект вне локальной среды.
3. Только после этого брать Workstream 3 и выбирать первый post-MVP релиз по продуктовым приоритетам.

## Milestones
- Milestone 1: `pnpm dev` запускается предсказуемо и сам объясняет проблемы с портами.
- Milestone 2: проект доступен по внешнему URL на сервере с рабочим логином, чатом и realtime.
- Milestone 3: выбран и описан первый post-MVP релизный батч.
- Milestone 4: первая post-MVP функция доведена до рабочего состояния и покрыта тестами.
