## Objective
Сделать пузырьки сообщений еще компактнее за счет переноса времени из отдельной нижней строки к правому краю bubble, как в вашем референсе, не меняя размер основного текста сообщения.

## Constraints
- Размер и стиль основного текста сообщения не меняем.
- Основная цель: убрать лишнюю высоту bubble, а не менять типографику.
- Не ломаем:
  - входящие сообщения
  - исходящие сообщения
  - длинные сообщения
  - сообщения с вложениями
  - текущие `data-testid`
- Меняем только frontend-верстку и связанную логику рендера.

## Current State
Сейчас в [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx):
- время рендерится отдельным `<p>` снизу bubble
- именно это добавляет лишнюю высоту, особенно у коротких text-only сообщений
- short text-only bubble уже ужат по `padding` и `radius`, но нижняя строка со временем все равно раздувает карточку
- для attachment-сообщений отдельная нижняя строка с временем пока логичнее и безопаснее

## Approach
Сделать 2 режима bubble:
- `inline-meta` режим для text-only сообщений, где время уходит вправо внутрь той же визуальной строки/нижнего края bubble
- текущий `stacked-meta` режим для attachment-сообщений и при необходимости для более сложных кейсов

Это даст максимальное уменьшение высоты там, где это реально нужно: на коротких обычных сообщениях.

## Plan

### Phase 1: Разделить сообщения на layout-варианты
- Явно выделить в компоненте варианты:
  - `text-only`
  - `text + attachments`
  - `attachments-only`
- Для `text-only` подготовить новый компактный layout с временем справа.
- Для `attachments` временно оставить более безопасную текущую вертикальную структуру.
- Output / exit criterion:
  - в коде есть понятное условие, какие сообщения получают inline timestamp, а какие нет

### Phase 2: Перестроить разметку text-only bubble
- Внутри bubble для `text-only` сообщений заменить схему:
  - `text`
  - `timestamp as separate paragraph`
  на схему:
  - `message content row`
  - внутри нее текст и время живут в одной горизонтальной компоновке
- Наиболее вероятная реализация:
  - `inline-flex items-end gap-2`
  - или компактный `grid` с двумя колонками: `text | time`
- Время должно визуально прижиматься к правой стенке bubble.
- Output / exit criterion:
  - отдельная нижняя строка со временем исчезает у text-only сообщений

### Phase 3: Сохранить читаемость коротких и длинных сообщений
- Проверить 3 сценария:
  - очень короткое сообщение: `a`, `ok`, `a?`
  - обычная короткая фраза
  - длинное сообщение с переносом на 2+ строки
- Для длинных сообщений решить, как должен вести себя timestamp:
  - оставаться справа в нижнем крае
  - не налезать на текст
  - не ломать переносы
- Если один layout не подходит всем случаям, разделить:
  - `short inline bubble`
  - `multiline inline bubble`
- Output / exit criterion:
  - время справа работает и на коротких, и на длинных text-only сообщениях без визуального конфликта

### Phase 4: Поджать геометрию bubble после переноса времени
- После переноса времени вправо еще раз уменьшить именно форму bubble:
  - вертикальные `padding`
  - при необходимости слегка горизонтальные `padding`
  - возможно еще немного `radius`
- Делать это уже после новой компоновки, потому что старая геометрия была под отдельную нижнюю строку.
- Output / exit criterion:
  - bubble становится визуально ближе к референсу и заметно ниже по высоте

### Phase 5: Проверить входящие сообщения с именем отправителя
- У входящих сообщений сверху есть имя отправителя.
- Нужно проверить, что после переноса времени вправо:
  - имя не конфликтует с нижним inline timestamp
  - bubble не выглядит перегруженным
- При необходимости отдельно подправить spacing для incoming variant.
- Output / exit criterion:
  - входящие сообщения сохраняют структуру, но тоже становятся компактнее

### Phase 6: Не сломать attachment-сообщения
- Для сообщений с вложениями не форсировать новый inline timestamp сразу, если это ухудшает layout.
- Оставить их на текущем stacked-режиме, если это безопаснее.
- Отдельно проверить:
  - image preview
  - file card
  - attachment-only message
- Output / exit criterion:
  - улучшение касается text-only сообщений, а attachment UI остается стабильным

### Phase 7: Довести визуальное выравнивание времени
- Проверить:
  - отступ между текстом и временем
  - выравнивание времени по нижней кромке
  - цвет/контраст времени на outgoing и incoming bubble
- Не менять размер текста сообщения, но при необходимости можно чуть скорректировать именно line-height блока времени, если это не меняет визуальный размер текста.
- Output / exit criterion:
  - timestamp выглядит естественно встроенным в bubble, а не “приклеенным сбоку”

## Testing and Validation
- Проверить исходящее короткое сообщение:
  - bubble заметно ниже
  - время стоит справа у стенки
- Проверить входящее короткое сообщение:
  - имя, текст и время не конфликтуют
- Проверить длинное text-only сообщение:
  - переносы корректны
  - время не налезает на текст
- Проверить несколько коротких сообщений подряд:
  - список сообщений выглядит плотнее и аккуратнее
- Проверить attachment message:
  - layout не сломан
- Проверить mobile width:
  - bubble не становится слишком узким
  - время не ломает переносы
- Прогнать `pnpm --filter @repo/web typecheck`, `pnpm verify:web`, `pnpm test:ui:e2e:auto`

## Risks and Dependencies
- Риск: время справа начнет конфликтовать с длинным текстом.
- Mitigation:
  - при необходимости разделить layout для single-line и multiline text-only сообщений.

- Риск: bubble станет слишком узким из-за текста плюс времени в одной строке.
- Mitigation:
  - использовать `inline-flex` или `grid` с корректным переносом текста и нижним выравниванием времени.

- Риск: attachment-сообщения визуально сломаются, если применить тот же layout ко всем.
- Mitigation:
  - ограничить новый режим только `text-only` сообщениями.

- Риск: входящие сообщения с именем отправителя будут выглядеть перегруженно.
- Mitigation:
  - отдельно проверить incoming layout и при необходимости чуть скорректировать spacing имени.
