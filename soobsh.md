## Objective
Сделать само облако сообщения еще компактнее по размеру, не меняя размер шрифта, размер текста и читаемость контента. Меняем только геометрию bubble: внутренние отступы, радиусы, минимальные размеры и расстояния вокруг времени.

## Assumptions
- Размер шрифта текста сообщения оставляем как есть.
- Размер шрифта времени тоже по возможности не трогаем, если это не потребуется для выравнивания.
- Изменяем только bubble-обертку и связанную с ней геометрию.
- Основной акцент: короткие сообщения должны выглядеть еще компактнее, почти без лишнего воздуха.
- Вложения и image preview не должны деградировать.

## Current State
Сейчас после последней правки в [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx):
- bubble уже стал компактнее
- текстовый `leading` сохранен комфортным
- расстояние между текстом и временем уменьшено
- короткие text-only сообщения уже сжаты, но пользователь хочет еще меньший bubble
- значит следующая итерация должна идти не через шрифт, а через:
  - еще более маленькие `py`
  - возможно чуть меньшие `px`
  - меньший radius
  - меньшую минимальную визуальную “массу” bubble

## Approach
Доработать bubble в 2 слоя:
- сначала еще сильнее сжать text-only bubble
- затем отдельно проверить incoming/outgoing и очень короткие сообщения
- если нужно, ввести супер-компактный режим только для очень коротких text-only сообщений

## Plan
### Phase 1: Зафиксировать, что именно нельзя трогать
- Явно оставить без изменений:
  - `text-sm`
  - текущий размер текста сообщения
  - базовую читаемость переносов
- Сосредоточиться только на:
  - `px`
  - `py`
  - `border-radius`
  - расстоянии до timestamp
  - возможной минимальной ширине/высоте bubble
- Output / exit criterion:
  - понятно, что следующая итерация меняет только форму облака, а не типографику

### Phase 2: Уменьшить внутренние отступы bubble еще сильнее
- В [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx) еще уменьшить:
  - вертикальные отступы у bubble
  - при необходимости и горизонтальные, но аккуратно
- Особенно для `text-only` сообщений:
  - оставить текст читаемым
  - убрать остаточный “воздух” вокруг одной короткой строки
- Проверить отдельно:
  - outgoing bubble
  - incoming bubble
- Output / exit criterion:
  - облако заметно меньше, но текст не ощущается зажатым

### Phase 3: Уменьшить радиус bubble
- Текущий rounded-стиль может визуально увеличивать облако
- Проверить, станет ли bubble визуально компактнее, если:
  - уменьшить общий `rounded`
  - особенно на short text-only messages
- Для incoming/outgoing сохранить единый визуальный язык, но убрать ощущение “большой капсулы”
- Output / exit criterion:
  - bubble выглядит меньше не только по пикселям, но и по восприятию формы

### Phase 4: Поджать блок времени без изменения шрифта
- Не менять размер шрифта времени, но:
  - уменьшить нижний padding bubble
  - еще немного подтянуть время к тексту
  - проверить line-height у timestamp
- Смысл: пузырек должен уменьшиться за счет геометрии, а не за счет уменьшения текста
- Output / exit criterion:
  - время остается читаемым, но не раздувает нижнюю часть облака

### Phase 5: Ввести “super-compact” режим для очень коротких text-only сообщений
- Если после базового сжатия bubble все еще кажется крупным на сообщениях типа:
  - `a`
  - `ok`
  - `a?`
  - `да`
- Тогда добавить отдельное условие:
  - short text-only message
  - без вложений
  - длина текста ниже выбранного порога
- Для такого режима можно сделать:
  - еще меньше `py`
  - чуть меньше `px`
  - чуть меньше radius
- При этом длинные сообщения оставить на более мягкой геометрии
- Output / exit criterion:
  - самые короткие сообщения выглядят действительно компактно и визуально ближе к вашему референсу

### Phase 6: Проверить вложения и входящие сообщения
- Убедиться, что новая компактность bubble не ломает:
  - сообщения с attachment
  - image preview
  - входящие сообщения с именем отправителя
- Возможно, итоговый режим будет такой:
  - `text-only` сообщения сильно уплотняем
  - `attachment` сообщения оставляем чуть свободнее
- Output / exit criterion:
  - компактность достигается именно там, где она нужна, без регресса в attachment layout

## Risks and Dependencies
- Риск: слишком маленькие `px/py` сделают bubble тесным и “сдавленным”.
- Mitigation:
  - сильнее сжимать только short text-only messages, а не все сообщения одинаково.

- Риск: уменьшение radius испортит характер интерфейса.
- Mitigation:
  - уменьшать radius умеренно и смотреть на итоговое восприятие, а не только на цифры.

- Риск: timestamp начнет визуально конфликтовать с текстом.
- Mitigation:
  - уменьшать геометрию bubble, но не сливать время с текстом в одну кучу.

- Риск: attachment bubble случайно станет слишком узким/маленьким.
- Mitigation:
  - применять более агрессивное сжатие только к text-only bubble.
