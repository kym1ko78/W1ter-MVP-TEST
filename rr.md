## Goal
Убрать постоянный шум в `git status` от tracked build-артефактов `apps/web/.next-e2e` и отдельно стабилизировать поведение `apps/web/next-env.d.ts`, чтобы рабочее дерево оставалось чистым после обычных запусков и проверок.

## Assumptions
- `.next-e2e` не должен храниться в репозитории как рабочий build-output.
- `apps/web/next-env.d.ts` нельзя автоматически считать “мусором”: для Next.js это служебный файл, и сначала нужно решить, должен ли он оставаться tracked.
- В репозитории уже есть `.gitignore`, где `apps/web/.next-e2e/` игнорируется, но это не помогает, потому что файлы уже отслеживаются Git.
- Цель задачи: не просто временно очистить текущее дерево, а убрать сам источник повторяющегося шума.

## Plan
### Phase 1: Зафиксировать точный состав шума
- Снять отдельный список tracked-файлов внутри `apps/web/.next-e2e`.
- Проверить, меняется ли только содержимое build-артефактов, или еще есть служебные generated-файлы вне этой папки.
- Отдельно зафиксировать, почему в `git status` появляется `apps/web/next-env.d.ts`:
  - реально ли меняется содержимое
  - или только переводы строк / regenerated header
  - или он был изменен вручную
- Output / exit criterion:
  - есть четкое разделение на:
    - build-артефакты `.next-e2e`
    - служебный файл `next-env.d.ts`

### Phase 2: Удалить `.next-e2e` из Git index без удаления из локальной работы
- Так как `.gitignore` уже содержит `apps/web/.next-e2e/`, нужно удалить эту папку именно из Git index, а не просто добавить ignore.
- Выполнить безопасную очистку отслеживания:
  - удалить `apps/web/.next-e2e` из index
  - оставить физические файлы локально до следующей сборки/перегенерации
- После этого проверить, что:
  - `git status` больше не показывает изменения внутри `.next-e2e`
  - новые билды не возвращают эту папку в tracked-состояние
- Output / exit criterion:
  - `.next-e2e` больше не отслеживается Git и перестает шуметь после прогонов

### Phase 3: Принять правильную политику для `next-env.d.ts`
- Проверить текущее содержимое `apps/web/next-env.d.ts`.
- Сравнить его с ожидаемым стандартным Next.js вариантом.
- Выбрать один из двух путей:
  - если файл должен оставаться в репозитории:
    - вернуть его к стабильному каноническому содержимому
    - исключить ручные/случайные изменения
  - если в этом проекте команда решит его не хранить:
    - добавить явное правило и перестроить процесс так, чтобы это не ломало `Next.js`
- Рекомендуемый путь:
  - оставить `next-env.d.ts` tracked, но привести к стабильному виду
- Output / exit criterion:
  - по `next-env.d.ts` есть осознанная политика, а не “временно скрыли проблему”

### Phase 4: Проверить сопутствующие generated-файлы
- Убедиться, что рядом не шумят другие уже tracked generated outputs:
  - `*.tsbuildinfo`
  - `.next`
  - временные кэши
- Проверить, не лежат ли такие файлы в индексе из старых коммитов.
- Если лежат:
  - оформить их отдельным cleanup-блоком в рамках той же задачи
  - не смешивать с source-файлами
- Output / exit criterion:
  - все build/cache artifacts либо игнорируются, либо осознанно tracked

### Phase 5: Проверить устойчивость после реального прогона
- После cleanup прогнать типичный цикл, который раньше засорял рабочее дерево:
  - `pnpm --filter @repo/web typecheck`
  - `pnpm verify:web`
  - при необходимости `pnpm test:ui:e2e:auto`
- Затем снова проверить `git status`.
- Цель:
  - после обычных web-checks дерево остается чистым
  - не появляются десятки файлов в `.next-e2e`
- Output / exit criterion:
  - проблема подтвержденно исчезает после реального сценария, а не только после ручной очистки

### Phase 6: Зафиксировать cleanup в документации и рабочем процессе
- Обновить `implementation-status.md` или краткую документацию, если это нужно для команды.
- Кратко зафиксировать:
  - что `.next-e2e` больше не tracked
  - что `next-env.d.ts` имеет стабильную политику
  - что считать нормальным/ненормальным шумом в `git status`
- Output / exit criterion:
  - у команды есть понятное правило, и проблема не вернется из-за повторного коммита артефактов

## Risks and Dependencies
- Риск: вместе с `.next-e2e` случайно убрать что-то, на что завязан pipeline.
- Mitigation:
  - сначала проверить, действительно ли это только generated output, а не проектные asset-файлы.

- Риск: `next-env.d.ts` ошибочно убрать из репозитория, хотя Next.js ожидает его наличие.
- Mitigation:
  - не игнорировать его автоматически; сначала проверить содержимое и принять явное решение.

- Риск: после cleanup артефакты снова попадут в индекс из-за `git add .`.
- Mitigation:
  - опираться на уже существующий `.gitignore` и убедиться, что tracked-связь действительно снята.

- Риск: в репозитории есть и другие tracked generated-файлы, которые продолжат шуметь даже после исправления `.next-e2e`.
- Mitigation:
  - отдельно проверить `*.tsbuildinfo` и соседние output-файлы в этой же задаче.

## Sequencing Rationale
- Сначала нужно разделить проблему на две разные части: `.next-e2e` и `next-env.d.ts`.
- Потом безопасно убрать из Git только явные build-артефакты.
- И только после этого решать судьбу `next-env.d.ts`, потому что это уже не мусорный кэш, а служебный Next-файл.
- Финальная проверка должна идти после реальных build/test сценариев, иначе cleanup может оказаться временным.
