## Goal
Убрать текущую ошибку локального прогона UI e2e на Windows: `EADDRINUSE :4000` и нестабильный запуск `Playwright`, чтобы тест запускался предсказуемо и без ручной путаницы.

## Assumptions
- `PostgreSQL` уже работает нормально.
- `API` собирается и умеет отвечать на `GET /health`.
- `web build` уже проходит.
- Главная проблема сейчас не в логике чата, а в способе запуска локальных серверов для `Playwright`.

## Plan
### Phase 1: Зафиксировать правильную модель запуска
Нужно выбрать один режим и больше их не смешивать.

Вариант A: ручной режим  
Подходит для Windows лучше всего.
- API поднимается вручную один раз.
- Web поднимается вручную один раз.
- `Playwright` запускается с `PLAYWRIGHT_SKIP_WEBSERVER=1`.

Вариант B: автоматический режим  
- `Playwright` сам поднимает `api` и `web`.
- Перед этим на `3000` и `4000` не должно быть уже запущенных процессов.

Рекомендация:
- для локальной отладки использовать Вариант A
- для будущей автоматизации оставить Вариант B

Output / exit criterion:
- есть один понятный режим запуска без двойного старта API

### Phase 2: Убрать конфликт порта `4000`
Текущая ошибка:
- `EADDRINUSE: address already in use :::4000`

Это значит:
- `API` уже запущен
- вы запускаете его второй раз поверх уже работающего процесса

Что делать:
- перед стартом API проверять:
```cmd
curl http://127.0.0.1:4000/health
```
Если видите:
```json
{"status":"ok"}
```
то API уже жив, повторно запускать его не нужно.

Если нужен чистый запуск:
```cmd
netstat -ano | findstr :4000
taskkill /PID <PID> /F
```

Output / exit criterion:
- порт `4000` либо свободен, либо вы осознанно используете уже запущенный API

### Phase 3: Зафиксировать правильный ручной запуск
Для ручного стабильного режима:

Терминал 1:
```cmd
cd /d C:\Users\User\Desktop\Project
pnpm --filter @repo/api build
pnpm --filter @repo/api start
```

Если `health` уже отвечает, этот терминал можно пропустить.

Терминал 2:
```cmd
cd /d C:\Users\User\Desktop\Project
pnpm --filter @repo/web build
pnpm --filter @repo/web exec next start --hostname 127.0.0.1 --port 3000
```

Терминал 3:
```cmd
cd /d C:\Users\User\Desktop\Project
set PLAYWRIGHT_SKIP_WEBSERVER=1
pnpm exec playwright test --config=playwright.config.cjs
```

Output / exit criterion:
- `Playwright` стартует, не пытаясь заново поднимать уже работающие серверы

### Phase 4: Проверить автоматический режим `Playwright`
После стабилизации ручного режима нужно проверить автоматический.

Что проверить:
- нет ли висящих процессов на `3000`
- нет ли висящих процессов на `4000`
- `playwright.config.cjs` использует корректные команды
- `Playwright` ждет:
  - `http://127.0.0.1:4000/health`
  - `http://127.0.0.1:3000/login`

Команда:
```cmd
cd /d C:\Users\User\Desktop\Project
pnpm test:ui:e2e
```

Если снова будет конфликт портов:
- значит для локальной Windows-среды автоматический режим пока хуже ручного

Output / exit criterion:
- либо автоматический режим проходит
- либо мы официально фиксируем manual-first режим как основной для Windows

### Phase 5: Проверить сам `Playwright`-сценарий
Если серверы поднялись, но тест все еще падает, тогда проблема уже в самом сценарии.

Что проверить:
- открывается ли `/register`
- проходит ли регистрация обоих пользователей
- создается ли чат
- доходит ли сообщение realtime
- нет ли таймаута на `goto` или `waitForURL`

Если падает:
- смотреть конкретный шаг в `tests/playwright/chat-flow.spec.ts`
- отдельно проверять:
  - `gotoAndWaitForTestId`
  - `registerUser`
  - ожидание `conversation-title`
  - ожидание `message-item`

Output / exit criterion:
- тест падает либо в bootstrap, либо в бизнес-логике, и мы точно знаем где

### Phase 6: Зафиксировать итоговый рабочий протокол
После проверки нужно закрепить один понятный способ работы.

Если ручной режим стабилен:
- записать в `README.md`
- записать в `implementation-status.md`
- указать, что на Windows локально UI e2e запускается через уже поднятые `api + web`

Если автоматический режим тоже стабилен:
- оставить `pnpm test:ui:e2e` как основной
- ручной режим оставить как fallback

Output / exit criterion:
- у проекта есть один документированный рабочий сценарий запуска UI e2e

## Risks and Dependencies
- Риск: пользователь запускает API дважды и снова получает `EADDRINUSE`.
- Митигация: всегда сначала проверять `curl http://127.0.0.1:4000/health`.

- Риск: `Playwright` будет конфликтовать с уже открытым `pnpm dev`.
- Митигация: не держать одновременно `pnpm dev` и `pnpm test:ui:e2e` в auto-mode.

- Риск: `localhost` и `127.0.0.1` дадут разное поведение на Windows.
- Митигация: использовать `127.0.0.1` в `Playwright` и локальных env.

- Риск: ошибка окажется не в серверах, а в самом тесте.
- Митигация: сначала стабилизировать старт серверов, потом уже анализировать шаг падения теста.
