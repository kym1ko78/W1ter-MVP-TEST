## Goal
Сделать сообщения еще компактнее по внутренним отступам, особенно короткие исходящие bubbles, не меняя размер текста и не ломая время/читаемость.

## Assumptions
- Меняем только frontend-верстку bubbles в [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx).
- Размер шрифта текста сообщения и времени желательно оставить тем же.
- Основной визуальный фокус — короткие self-messages вроде `д`, `ок`, `а`, где даже текущие `px-3 py-1` все еще кажутся великоватыми.
- Удалять/переносить кнопку `Удалить` сейчас не нужно; меняется только сам bubble и внутренний контент.

## Constraints
- Не ломать layout длинных сообщений.
- Не ломать attachment messages.
- Не ухудшить читаемость времени внутри bubble.
- Не менять существующую логику inline timestamp, если это не нужно для компактности.

## Architecture / Approach
Уплотнение нужно делать не глобально “всем сообщениям одинаково”, а с акцентом на:
- short text-only bubbles
- compact text-only bubbles

Потому что именно на коротких сообщениях лишний padding заметнее всего.

Решение должно идти через:
- уменьшение `px/py`
- уменьшение `gap-x` между текстом и временем
- при необходимости небольшое уменьшение `rounded`
- но без изменения font-size

- Output / exit criterion:

### Phase 2: Уменьшить short text-only bubble
- Для самого компактного случая уменьшить:
  - `px`
  - `py`
  - при необходимости `rounded`
- Цель:
  - короткие сообщения должны выглядеть почти “собранной капсулой”, а не большим пузырем
- Проверить сообщения длиной 1-3 символа.
- Output / exit criterion:
  - короткие self-bubbles заметно компактнее визуально

### Phase 3: Уменьшить compact text-only bubble
- Для `compactBubble` тоже слегка уменьшить:
  - `px`
  - `py`
  - возможно `rounded`
- Не так агрессивно, как для `shortTextOnlyBubble`, чтобы обычные короткие фразы не стали тесными.
- Output / exit criterion:
  - не только супер-короткие, но и обычные короткие сообщения выглядят плотнее

### Phase 4: Поджать внутреннюю компоновку текста и времени
- В inline layout проверить:
  - `gap-x-1.5`
  - `pb-0.5` у timestamp
- Если нужно:
  - уменьшить горизонтальный gap между текстом и временем
  - немного уменьшить нижнюю микрокоррекцию времени
- Цель:
  - уменьшить визуальную ширину bubble без изменения шрифта
- Output / exit criterion:
  - текст и время сидят плотнее, но не слипаются

### Phase 5: Проверить длинные и многострочные сообщения
- После уплотнения коротких сообщений проверить, не пострадал ли обычный text-only layout:
  - длинные сообщения
  - многострочные сообщения
- При необходимости оставить более агрессивное уплотнение только для short case, а для остальных — мягкое.
- Output / exit criterion:
  - длинные bubbles не становятся тесными и не теряют читаемость

### Phase 6: Проверить incoming и outgoing баланс
- Хотя пример касается self-messages, нужно проверить, как те же изменения смотрятся на incoming bubble.
- Возможно:
  - одинаковые классы останутся ок
  - или короткие incoming messages тоже выиграют от тех же значений
- Output / exit criterion:
  - интерфейс остается цельным для обеих сторон переписки

### Phase 7: Проверить attachment messages отдельно
- Убедиться, что правка не затронула сообщения с вложениями:
  - image preview
  - file card
  - text + attachment
- Если changes затрагивают только text-only bubbles, attachment layout останется стабильным.
- Output / exit criterion:
  - вложения не деградируют

### Phase 8: Финальная визуальная полировка
- После базового уплотнения еще раз оценить:
  - не слишком ли велик внутренний отступ у очень коротких сообщений
  - не слишком ли крупный радиус относительно уменьшенного пузыря
- При необходимости сделать одну последнюю микроитерацию:
  - `px`
  - `py`
  - `rounded`
  - `gap-x`
- Output / exit criterion:
  - bubble выглядит компактно и чисто, без ощущения “перетянутой” верстки

## Risks and Dependencies
- Риск: слишком сильное уменьшение `px/py` сделает bubble визуально “зажатым”.
- Mitigation:
  - сильнее ужимать только `shortTextOnlyBubble`, а не все сообщения одинаково.

- Риск: время начнет конфликтовать с текстом.
- Mitigation:
  - уменьшать `gap-x` аккуратно, без слияния элементов.

- Риск: уменьшенный `rounded` испортит визуальный язык интерфейса.
- Mitigation:
  - трогать radius мягко и только если после уменьшения padding он реально кажется слишком большим.
