## Objective
Сделать поле ввода сообщения по умолчанию такой же высоты, как кнопка `Отправить`, при этом сохранить текущее поведение увеличения поля по высоте.

## Assumptions
- Под “пусть он увеличивается так же” понимается: `textarea` остается растягиваемой по высоте, но ее стартовая высота становится равной кнопке.
- Меняется только UI composer, без изменения backend и логики отправки.
- Счетчик `0/4000` должен остаться, но не должен ломать визуальное выравнивание поля и кнопки.

## Current State
Сейчас в `apps/web/components/conversation-view.tsx`:
- `textarea` имеет `rows={3}` и `min-h-[72px]`
- кнопка живет в отдельной колонке `sm:w-[180px]`
- счетчик символов находится под кнопкой
- из-за этого поле ввода визуально выше кнопки уже в дефолтном состоянии

## Implementation Plan

### Phase 1: Зафиксировать целевую геометрию composer
- Выбрать одну явную высоту для базового состояния контролов, например `56px` или `60px`.
- Сделать эту высоту общей для:
- поля ввода
- кнопки `Отправить`
- Вынести это в понятный CSS-класс или прямо в Tailwind-классы, чтобы не было “магических” разных размеров в двух местах.

Output / exit criterion:
- есть одно согласованное значение высоты для обоих элементов

### Phase 2: Перестроить layout composer
- Пересобрать нижний блок composer так, чтобы:
- в первой строке стояли `textarea` и кнопка
- во второй строке стоял счетчик символов
- Убрать зависимость высоты кнопки от блока со счетчиком.
- На desktop использовать layout вида:
- `input | button`
- ниже `counter`
- На mobile сохранить удобный стек, но так, чтобы дефолтная высота поля все равно была равна кнопке.

Output / exit criterion:
- счетчик больше не влияет на визуальное сравнение высоты поля и кнопки

### Phase 3: Изменить стартовое состояние textarea
- Заменить `rows={3}` на `rows={1}`.
- Задать `textarea`:
- стартовую высоту = высоте кнопки
- `min-height` = той же высоте
- Проверить внутренние `padding-top` / `padding-bottom`, чтобы текст был визуально по центру.
- Сохранить `placeholder` и текущую логику ошибок/отправки.

Output / exit criterion:
- при пустом поле `textarea` и кнопка визуально одной высоты

### Phase 4: Сохранить поведение увеличения по высоте
- Если оставляем текущее ручное растягивание:
- не убирать `textarea`
- разрешить вертикальный resize
- убедиться, что после ручного увеличения кнопка остается рядом и интерфейс не ломается
- Если позже захотите авто-рост по вводу, это можно включить здесь же:
- добавить `ref`
- на `onChange` сбрасывать высоту и выставлять `scrollHeight`
- ограничить максимальную высоту, после которой включается внутренний скролл

Output / exit criterion:
- поле стартует как кнопка, но продолжает увеличиваться по высоте без поломки layout

### Phase 5: Подчистить визуальные детали
- Проверить радиусы, чтобы кнопка и поле выглядели как один аккуратный composer-блок.
- Проверить расстояние между полем, кнопкой и счетчиком.
- Убедиться, что ручка resize визуально не конфликтует с рамкой.
- Проверить, не прыгает ли счетчик при наборе текста.

Output / exit criterion:
- composer выглядит цельно и аккуратно в пустом и заполненном состояниях

## Risks and Dependencies
- Риск: счетчик под кнопкой снова визуально исказит высоту controls.
- Митигация: вынести счетчик в отдельную строку layout.
- Риск: `rows={1}` + текущие `padding` дадут слишком маленькое или слишком большое поле.
- Митигация: подгонять через одну фиксированную высоту, а не только через `rows`.
- Риск: ручной resize на mobile/desktop будет выглядеть по-разному.
- Митигация: проверить оба режима и при необходимости ограничить `resize-y`.
