Качественно доработать чат-интерфейс в шести направлениях: сделать трубку чуть уже, добавить анимацию нажатия на чаты, внедрить отдельную анимацию открытия профиля, добавить индивидуальные `#`-ники, полностью переработать flow удаления сообщений, а также приблизить отступы сообщений к телеграм-подобной геометрии.

## Constraints
- Нельзя потерять уже возвращенные фичи: reply, edit, forward, reactions, media attachments, realtime presence/typing, профильную панель и обновленный overlay-scroll.
- Нужно устранить пересечение модалок: пока одно диалоговое окно открыто, интерфейс под ним не должен вызывать второе окно или выполнять конкурирующие destructive-action.
- Новый delete-flow должен работать и с обычным кликом по действию, и с правой кнопкой мыши по сообщению, если это финально выбранный trigger.
- Изменение геометрии сообщений не должно ломать поведение поиска, reply-preview, inline-meta, media-bubbles и hover-actions.

## Architecture / Approach
### 1. Разделить задачи на UI-only и full-stack
- UI-only:
  - сделать трубку уже
  - анимация нажатия на чаты
  - анимация открытия профиля
  - обновить отступы сообщения от стенок
  - заблокировать взаимодействие с остальным UI, пока открыт destructive modal
- Full-stack:
  - индивидуальные `#`-ники
  - режимы удаления сообщения: “у всех” / “у себя”

### 2. Перевести destructive-flow на единый modal-layer
- Вместо нескольких независимых `ConfirmDialog` в одном экране нужен централизованный слой управления destructive-окнами.
- В любой момент времени должен существовать только один активный destructive intent:
  - delete message
  - leave group
  - delete chat
  - remove member
- Это исключит ситуацию, когда на фоне одного окна открывается второе.

### 3. Вынести delete-message UX в отдельный сценарий
- Удаление сообщения больше не должно быть привязано только к маленькой inline-кнопке.
- Нужен контекстный вход:
  - правый клик по сообщению
  - fallback для touch/обычного UX, если потребуется
- После вызова должно открываться одно модальное окно по центру экрана с вариантами удаления.

## Risks and Dependencies
- `Удалить у себя` нельзя сделать качественно без изменения модели данных.
  Mitigation: заранее выбрать персистентную server-side модель скрытия сообщений, а не пытаться решить это только через фронт.

- Добавление `username` затронет Prisma, API, фронтовые типы, профиль и поиск.
  Mitigation: проводить это как отдельный вертикальный срез с миграцией и совместимой выдачей API.

- Контекстное меню по правому клику может конфликтовать с мобильными устройствами и touch UX.
  Mitigation: держать fallback-вход через обычное действие или long-press как следующий шаг, если потребуется.

- Централизация модалок может затронуть уже существующие confirm-flows удаления чата и участников.
  Mitigation: переводить все destructive-dialog на единый state-model за один проход, а не частично.

- Изменение отступов bubble может повлиять на hover-actions, reply-preview и media attachments.
  Mitigation: проверять геометрию для нескольких типов сообщений, а не только для короткого текста.

## Sequencing Rationale
- Сначала нужно стабилизировать modal-layer и delete-flow, потому что сейчас это behavioral bug и риск для destructive-actions.
- Потом уже имеет смысл встраивать два режима удаления на backend, иначе UI-выбор будет фиктивным.
- `#ники` логично делать отдельным full-stack блоком после delete-flow, потому что это независимая data-model задача.
- Анимации и геометрию сообщений лучше добивать после стабилизации поведения, чтобы визуальные правки не маскировали функциональные баги.
