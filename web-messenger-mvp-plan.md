# План MVP веб-мессенджера

## Цель
Собрать web-first MVP мессенджера на стеке `Next.js + React + TypeScript + NestJS + Socket.IO + PostgreSQL + Prisma`, в котором два пользователя могут зарегистрироваться, войти, найти или создать личный чат, обмениваться текстовыми сообщениями в реальном времени, видеть историю сообщений и пользоваться приложением с десктопа и мобильного браузера.

## Формат MVP

### Что входит в MVP
- Регистрация и вход по `email + password`
- Профиль пользователя: имя, аватар по желанию
- Личные чаты `1 на 1`
- Список чатов
- История сообщений
- Отправка и получение текстовых сообщений
- Доставка новых сообщений в реальном времени
- Непрочитанные сообщения
- Базовый статус `online / last seen`
- Адаптивный интерфейс для мобильного браузера
- Базовые ошибки, reconnect, пустые состояния

### Что не входит в MVP
- End-to-end encryption
- Телефонная авторизация через SMS
- Групповые чаты
- Каналы
- Боты
- Голосовые и видеозвонки
- Redis на старте
- Микросервисы
- Full-text поиск
- Стикеры, реакции, истории

## Успех MVP
- Пользователь может зарегистрироваться и повторно войти без ручных манипуляций с БД
- Два пользователя могут открыть чат в двух браузерах и видеть сообщения почти мгновенно
- История чата сохраняется после перезагрузки страницы
- Непрочитанные сообщения и `last seen` ведут себя предсказуемо
- Интерфейс usable на экране телефона и ноутбука
- Приложение можно поднять локально одной командой и развернуть в одном окружении без переписывания архитектуры

## Предположения
- Проект делает один разработчик или маленькая команда `1-3 человека`
- Первая версия рассчитана на малую нагрузку: до нескольких сотен активных пользователей
- На старте нужен только веб-клиент
- Для упрощения MVP используем `email + password`, а не телефон и SMS
- Отправку сообщения делаем через HTTP API, а Socket.IO используем для событий и мгновенной доставки на клиенты
- Локально PostgreSQL поднимается через Docker, а Redis откладываем до этапа масштабирования
- Разработка идет на Windows, но рабочую папку для проекта лучше держать в пути без кириллицы и пробелов, например `C:\dev\web-messenger`, чтобы избежать проблем с Docker volume mounts, CLI-инструментами и отдельными npm-пакетами

## Архитектурный подход для MVP

### Общая схема
- Frontend ходит в backend по `HTTP` за авторизацией, списком чатов, историей и отправкой сообщений
- Frontend держит отдельное `Socket.IO` соединение для realtime-событий
- Backend после успешной записи сообщения в PostgreSQL публикует событие нужным пользователям через Socket.IO
- Presence хранится в памяти одного backend-процесса
- Если позже backend станет больше одного экземпляра, тогда добавляется `Redis adapter` для Socket.IO

### Почему сообщение лучше отправлять через HTTP, а не сразу через сокет
- проще дебажить
- проще логировать и тестировать
- проще контролировать транзакцию записи в БД
- WebSocket остается каналом доставки событий, а не местом для всей бизнес-логики

## Модель данных MVP

### Таблицы

#### `users`
- `id`
- `email`
- `password_hash`
- `display_name`
- `avatar_url`
- `created_at`
- `updated_at`
- `last_seen_at`

#### `refresh_tokens`
- `id`
- `user_id`
- `token_hash`
- `expires_at`
- `created_at`
- `revoked_at`
- `user_agent`
- `ip_address`

#### `chats`
- `id`
- `type` со значением `direct`
- `created_at`
- `updated_at`
- `last_message_id`

#### `chat_members`
- `chat_id`
- `user_id`
- `joined_at`
- `last_read_message_id`

#### `messages`
- `id`
- `chat_id`
- `sender_id`
- `body`
- `created_at`
- `updated_at`
- `deleted_at` по желанию, можно не включать в первую итерацию

### Принципы модели
- В первой версии используйте `UUID` для первичных ключей
- Для MVP не нужен отдельный `message_reads` table, если непрочитанные считаются через `last_read_message_id`
- Вложения лучше вынести в отдельную таблицу только после стабилизации текстового чата

## Контракты API

### Auth
- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/logout`
- `POST /auth/refresh`
- `GET /auth/me`

### Users
- `GET /users/me`
- `GET /users/search?query=...`
- `PATCH /users/me`

### Chats
- `GET /chats`
- `POST /chats/direct`
- `GET /chats/:chatId`
- `GET /chats/:chatId/messages?cursor=...`
- `POST /chats/:chatId/messages`
- `POST /chats/:chatId/read`

### Socket.IO события

#### Клиент подписывается
- `connect`
- `join_user_room`
- `join_chat_room`

#### Сервер отправляет
- `message:new`
- `message:ack`
- `chat:updated`
- `chat:read`
- `presence:changed`

### Что лучше не делать в первой версии сокетов
- сложный event protocol
- доставку сообщения только через сокет без HTTP fallback
- межсерверный fan-out без Redis

## Что использовать в production для первой версии

### Рекомендованный путь
- Один Linux VPS
- Docker Compose
- Один контейнер `web`
- Один контейнер `api`
- Один контейнер `postgres`
- Reverse proxy через `Nginx` позже или сразу, если нужен единый домен и HTTPS

### Почему так
- проще деплой и отладка
- websockets работают предсказуемо
- не появляется ранняя сложность managed multi-service инфраструктуры

### Когда переходить к Redis
- когда у backend будет больше одного экземпляра
- когда нужен общий presence между инстансами
- когда Socket.IO надо масштабировать горизонтально

## Зависимости и риски

### Риск 1. Слишком большой scope
- Симптом: начинаете добавлять группы, файлы, звонки и телефонную авторизацию до запуска
- Как избежать: заморозить scope MVP и держать backlog отдельно

### Риск 2. Сложная авторизация
- Симптом: много времени уходит на refresh flow и CORS/cookie issues
- Как избежать: заранее решить схему токенов, домены и cookie policy

### Риск 3. Нестабильный realtime
- Симптом: сообщения приходят, но UI периодически рассинхронизируется
- Как избежать: источник истины — PostgreSQL, а сокет только канал доставки событий

### Риск 4. Раннее масштабирование
- Симптом: добавляются Redis, очереди и сложный DevOps еще до первых пользователей
- Как избежать: один backend-инстанс до подтвержденной потребности

### Риск 5. Windows и путь проекта
- Симптом: странные проблемы с Docker, volume mounts и отдельными пакетами
- Как избежать: вести сам проект в ASCII-пути вроде `C:\dev\web-messenger`
