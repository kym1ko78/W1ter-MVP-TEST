## Goal
Переделать боковую панель меню в светлый цельный drawer: вся панель должна стать белой, а блок `Действия` должен перестать выглядеть как отдельное окно/карточка и превратиться в сплошной список кнопок внутри общей поверхности.

## Assumptions
- Речь идет о левом drawer/sidebar menu, который открывается по кнопке с тремя полосками.
- Меняем прежде всего визуальную структуру и layout, а не бизнес-логику пунктов меню.
- Пункты `Мой профиль`, `Выйти из аккаунта`, `Создать группу` и соседние действия должны остаться рабочими.
- Под “сплошными кнопками” понимается список row-элементов без отдельной внутренней floating-card оболочки.

## Plan
### Phase 1: Найти текущий источник панели
- Открыть компонент, который рендерит боковую панель меню в `chat-shell.tsx`.
- Найти:
- контейнер самого drawer;
- верхний профильный блок;
- секцию `Действия`;
- текущую обертку-карточку, которая делает этот блок “отдельным окном”.
- Зафиксировать, какие классы сейчас задают:
- общий фон панели;
- border/shadow;
- скругления внутренней карточки;
- spacing между действиями.
- Выходной результат фазы: ясная карта того, что нужно сделать белым и что нужно “сплющить” в единый список.

### Phase 2: Сделать весь drawer белой общей поверхностью
- Перевести фон всей боковой панели в светлый цвет из текущей системы проекта.
- Убедиться, что profile header, close area и остальная часть drawer визуально принадлежат одной белой поверхности.
- Сохранить разделение через:
- тонкие бордеры;
- отступы;
- мягкие нейтральные тени только если они уже нужны внешнему drawer.
- Убрать ощущение “темной панели с белыми карточками” или “белой панели с отдельными островками”, если такое осталось.
- Выходной результат фазы: вся панель читается как единый белый sheet.

### Phase 3: Превратить `Действия` в сплошной список
- Убрать у блока `Действия` отдельную карточную оболочку, если она сейчас есть:
- внутренний белый фон;
- отдельная рамка;
- отдельное скругление как у floating-card.
- Перестроить действия в список row-кнопок внутри общей панели:
- одинаковая высота строк;
- иконка слева;
- текст справа;
- единый hover;
- тонкие разделители или отступы между строками по необходимости.
- Убедиться, что список выглядит цельным и спокойным, как на референсе, а не как набор отдельных карточек.
- Выходной результат фазы: `Действия` выглядят как встроенная часть drawer, а не как вложенное окно.

### Phase 4: Согласовать верхний профильный блок с новым drawer
- Проверить верхнюю часть меню:
- avatar;
- `MESSENGER`;
- имя;
- email;
- `Открыть профиль`;
- кнопка закрытия.
- Довести этот блок до той же белой системы, чтобы он не спорил с обновленным списком действий.
- Если нужно, скорректировать:
- нижнюю границу между header и списком;
- типографику меток;
- muted text;
- hover для `Открыть профиль`.
- Выходной результат фазы: header панели и список действий ощущаются как один модуль.

### Phase 5: Проверить поведение кнопок после визуальной перестройки
- Убедиться, что после удаления внутренней карточки не сломались:
- переход в профиль;
- logout;
- открытие `Создать группу`;
- остальные пункты меню.
- Проверить, что click-area кнопок не стала слишком маленькой или, наоборот, конфликтующей между строками.
- Выходной результат фазы: весь список по-прежнему рабочий, несмотря на смену layout.

### Phase 6: Визуальная полировка
- Проверить, не нужно ли добавить:
- легкие разделители между строками;
- единый `rounded` только у внешнего drawer, а не у внутренних строк;
- одинаковые вертикальные ритмы между секциями.
- Если внутри панели есть другие блоки ниже `Действия`, привести их к тому же языку:
- не отдельные карточки в карточке;
- а секции внутри одного белого полотна.
- Выходной результат фазы: панель выглядит цельно и без “вложенных окон”.

### Phase 7: Validation
- Прогнать `pnpm --filter @repo/web typecheck`.
- Прогнать `pnpm verify:web`.
- Вручную проверить:
- открытие/закрытие панели;
- обычный desktop layout;
- узкий экран;
- hover/active состояния строк;
- переходы и logout;
- открытие режима создания группы.
- Выходной результат фазы: панель визуально обновлена и функционально стабильна.

## Risks and Dependencies
- Риск: можно убрать внутреннюю карточку, но потерять визуальную структуру секции `Действия`.
- Смягчение: оставить секционный заголовок, единый spacing и аккуратные row-hover состояния.

- Риск: белая панель без вложенных карточек станет слишком “плоской”.
- Смягчение: использовать типографику, тонкие границы и четкий вертикальный ритм вместо лишних рамок.

- Риск: действия визуально сольются с profile header.
- Смягчение: добавить ясную секционную границу между header и списком действий.
