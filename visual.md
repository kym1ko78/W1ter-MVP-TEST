# План удаления внешних отступов и краев на auth-экране

## Goal
Убрать внешние расстояния между краями окна и auth-интерфейсом, чтобы экран логина/регистрации лежал плотнее к границам viewport и не выглядел как отдельная карточка с большими полями по краям.

## Assumptions
- Речь идет о внешней auth-сцене, а не о внутренних отступах самих form blocks.
- Нужно убрать именно пространство между "стенами" окна и внешним контейнером auth-layout.
- Внутренние отступы в самой форме и контентных блоках можно оставить, если они не мешают.
- По смыслу это та же задача, что уже делалась для chat shell: убрать card-like outer framing.

## Current State
Сейчас auth-screen выглядит как большой белый canvas внутри внешней сцены:
- у `main.auth-scene` есть внешние `px` / `py`
- у центрального контейнера есть `max-width`, `rounded` и card-like framing
- из-за этого между границами экрана и интерфейсом остается заметное пустое пространство
- визуально auth-layout ощущается как отдельная карточка внутри еще одной сцены

## Approach
Убрать внешний framing auth-screen на уровне:
1. `main.auth-scene`
2. корневого auth-контейнера
3. при необходимости радиуса/рамки внешней оболочки

При этом сохранить:
- структуру двух колонок
- внутренние отступы форм
- читабельность на desktop и mobile

## Plan

### Phase 1: Найти внешний контейнер auth-layout
- Проверить `apps/web/components/auth-screen.tsx`.
- Найти элементы, которые создают внешние поля:
  - `main.auth-scene`
  - `mx-auto`
  - `max-w-[1240px]`
  - `px-*` / `py-*`
  - `rounded-[38px]`
  - border/shadow у внешней оболочки
- Output / exit criterion:
  - точно определены все классы, создающие расстояние между экраном и auth-интерфейсом

### Phase 2: Убрать внешние padding у auth-scene
- Снять или сильно уменьшить `px` / `py` у корневого `main`.
- Убедиться, что auth-layout действительно подходит ближе к краям viewport.
- Проверить desktop и mobile отдельно.
- Output / exit criterion:
  - основной внешний зазор по периметру экрана уменьшен или убран

### Phase 3: Убрать card-frame у внешнего белого контейнера
- Для главной обертки auth-layout пересмотреть:
  - `max-width`
  - `mx-auto`
  - `rounded-*`
  - outer border
  - outer shadow
- Если задача именно “без расстояния между стенами и интерфейсом”, внешний контейнер должен стать:
  - шире
  - ближе к full-width/full-height
  - без ощущения отдельной карточки
- Output / exit criterion:
  - auth-layout больше не выглядит как плавающая карточка внутри страницы

### Phase 4: Выпрямить внешние края при необходимости
- Если после снятия внешних отступов большие радиусы внешней оболочки начинают мешать, убрать или уменьшить их.
- Важно отличать:
  - внешние края большого auth-shell
  - внутренние углы у form card и внутренних блоков
- Рекомендация:
  - наружную оболочку сделать ровнее
  - внутренние карточки можно оставить мягкими
- Output / exit criterion:
  - внешний контур auth-screen становится ближе к краям и выглядит более цельно

### Phase 5: Проверить внутренний баланс после растяжения
- После растяжения auth-screen убедиться, что:
  - левый информационный блок не “разъезжается” слишком сильно
  - правая темная колонка не теряет композицию
  - форма входа не уезжает и не становится слишком мелкой на фоне большого холста
- При необходимости локально подправить ширины внутренних колонок и их padding.
- Output / exit criterion:
  - экран занимает больше пространства, но композиция остается сильной

### Phase 6: Проверить mobile и tablet
- На узких экранах убедиться, что снятие внешних полей не ухудшает UX.
- При необходимости оставить адаптивный компромисс:
  - desktop почти edge-to-edge
  - mobile с минимальным безопасным padding
- Output / exit criterion:
  - решение хорошо выглядит не только на desktop

### Phase 7: Финальная визуальная проверка
- Проверить `login`
- Проверить `register`
- Сравнить до/после:
  - стало ли меньше пустого пространства по краям
  - исчезло ли ощущение "карточки внутри еще одной карточки"
- Output / exit criterion:
  - auth-screen визуально плотнее прилегает к экрану и выглядит современнее

## Testing and Validation

- Проверить desktop layout
- Проверить mobile width
- Прогнать:
  - `pnpm --filter @repo/web typecheck`
  - `pnpm verify:web`

## Risks and Dependencies
- Риск: при полном снятии внешних отступов экран станет слишком "прилипшим" на mobile.
- Mitigation:
  - оставить небольшой adaptive padding для маленьких экранов.

- Риск: внешний контейнер без радиуса сломает визуальный баланс с внутренними блоками.
- Mitigation:
  - убирать только внешний framing, не трогая внутренние карточки без необходимости.

- Риск: слишком широкое растяжение испортит композицию двух колонок.
- Mitigation:
  - после снятия outer spacing отдельно проверить пропорции колонок.
