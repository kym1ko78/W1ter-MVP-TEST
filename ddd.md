## Goal
Сделать так, чтобы после отправки сообщения чат автоматически прокручивался вниз к последнему сообщению.

## Assumptions
- Изменения нужны только для frontend-поведения в [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx).
- Автоскролл нужен именно после успешной локальной отправки сообщения текущим пользователем.
- Нельзя ломать обычную ручную прокрутку истории.
- Желательно, чтобы решение не конфликтовало с уже существующим realtime-обновлением списка сообщений.

## Constraints
- Не прокручивать агрессивно при каждом входящем сообщении без необходимости.
- Не ломать attachment-only отправку.
- Не делать автоскролл до того, как сообщение реально появилось в списке.
- Сохранить текущие проверки `typecheck`, `verify:web`, `UI e2e`.

## Architecture / Approach
Лучший путь:
- добавить ref на scroll-контейнер списка сообщений
- после успешной отправки инициировать прокрутку вниз
- запускать это не “по submit”, а после фактического появления нового сообщения в render list / query cache

Это надежнее, чем скроллить сразу в `handleSubmit`, потому что DOM еще может не успеть обновиться.

## Plan
### Phase 1: Зафиксировать точку скролла
- Найти контейнер списка сообщений в [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx):
  - блок с `data-testid="message-list"`
- Добавить для него отдельный `ref`.
- Решить, будем ли скроллить:
  - сам контейнер через `scrollTop = scrollHeight`
  - или к нижнему sentinel-элементу через `scrollIntoView`
- Рекомендуемый вариант:
  - нижний sentinel, потому что он обычно устойчивее к изменениям layout
- Output / exit criterion:
  - в компоненте есть понятная опорная точка, к которой можно прокручивать вниз

### Phase 2: Отделить локальную отправку от остальных обновлений списка
- Так как список сообщений обновляется и от локальной отправки, и от realtime, нужно ввести явный триггер:
  - например флаг “ожидаем автоскролл после отправки”
- Этот флаг должен включаться:
  - при успешной локальной отправке
- И потребляться:
  - когда сообщение уже попало в DOM/список
- Output / exit criterion:
  - автоскролл запускается именно после собственного send-flow, а не на любое обновление списка

### Phase 3: Реализовать прокрутку после успешной отправки
- В `onSuccess` у `sendMessageMutation` или в связанном `useEffect` отметить, что нужен scroll-to-bottom.
- После обновления `messageItems`/`renderItems` выполнить реальную прокрутку вниз.
- Если используем sentinel:
  - вызвать `scrollIntoView({ block: "end" })`
- Если используем контейнер:
  - установить `scrollTop = scrollHeight`
- При необходимости использовать:
  - `requestAnimationFrame`
  - `queueMicrotask`
  - или короткий post-render эффект
  чтобы дождаться фактического обновления DOM
- Output / exit criterion:
  - после отправки сообщение появляется и экран доезжает до самого низа

### Phase 4: Настроить поведение для текста и вложений
- Проверить, что одинаково работают:
  - обычное текстовое сообщение
  - многострочное сообщение
  - сообщение с файлом
  - сообщение только с файлом
- Особенно важно для attachment preview / upload flow:
  - не скроллить слишком рано, пока сообщение еще не появилось в истории
- Output / exit criterion:
  - автоскролл стабилен для всех вариантов отправки

### Phase 5: Не ломать ручной просмотр истории
- Убедиться, что автоскролл не триггерится:
  - при простом чтении старых сообщений
  - при входящих сообщениях, если это не было явно задумано
- Если понадобится, ограничить логику только локальной отправкой.
- Это важно, чтобы пользователь мог спокойно читать старую часть истории без резких прыжков.
- Output / exit criterion:
  - ручной scroll вверх остается нормальным, автоскролл включается только в нужный момент

### Phase 6: Подумать о плавности прокрутки
- Решить, как должна вести себя прокрутка:
  - мгновенно
  - или мягко
- Для чата обычно приятнее мягкая, но не слишком медленная прокрутка.
- Если smooth-scroll начнет мешать тестам или ощущаться “тормозно”, оставить instant-scroll.
- Output / exit criterion:
  - выбран предсказуемый и стабильный scroll behavior

### Phase 7: Проверить совместимость с существующим realtime и read-flow
- Убедиться, что новая логика не конфликтует с:
  - mark-as-read effect
  - query invalidation
  - optimistic append / dedupe cache flow
- Проверить, что если сообщение пришло и тут же read-flow обновился, автоскролл не срабатывает повторно без необходимости.
- Output / exit criterion:
  - новый scroll behavior не ломает существующий message lifecycle

### Phase 8: Обновить тесты при необходимости
- Проверить, нужен ли апдейт [chat-flow.spec.ts](C:\Users\User\Desktop\Project\tests\playwright\chat-flow.spec.ts).
- Возможно текущий тест и так пройдет, но если нужно, можно явно проверять, что отправленное сообщение появилось внизу списка.
- Минимальная цель:
  - не сломать существующий e2e flow
- Output / exit criterion:
  - тесты проходят с новым scroll behavior

## Risks and Dependencies
- Риск: скролл будет вызываться слишком рано, до появления сообщения в DOM.
- Mitigation:
  - запускать его после обновления списка сообщений, а не сразу на submit.

- Риск: автоскролл начнет срабатывать и на входящие realtime-сообщения.
- Mitigation:
  - отделить локальную отправку через отдельный флаг.

- Риск: smooth-scroll ухудшит стабильность e2e.
- Mitigation:
  - при необходимости оставить instant-scroll или сделать поведение детерминированным.

- Риск: attachment flow имеет другую временную динамику из-за upload.
- Mitigation:
  - скроллить только после `sendMessageMutation.onSuccess`, а не в момент выбора файла.
