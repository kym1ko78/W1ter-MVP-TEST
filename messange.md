## Objective
Сделать пузырьки сообщений визуально компактнее: уменьшить лишний нижний отступ, сократить высоту карточки сообщения и приблизить отображение ко второму примеру, где текст и время собраны плотнее.

## Assumptions
- Меняем только frontend-верстку сообщений.
- Backend, realtime и логика отправки не затрагиваются.
- Цель не “сжать всё максимально”, а убрать ощущение пустоты снизу у bubble.
- Нужно сохранить читаемость и корректное поведение для:
  - коротких сообщений
  - длинных сообщений
  - входящих сообщений
  - исходящих сообщений
  - сообщений с вложениями

## Current State
Сейчас в [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx):
- пузырек сообщения использует `px-4 py-3`
- текст сообщения рендерится с `text-sm leading-6`
- время рендерится отдельным блоком с `mt-2`
- у входящих сообщений сверху еще есть имя отправителя с `mb-1`
- из-за сочетания `py-3`, `leading-6` и отдельного `mt-2` нижняя часть bubble выглядит слишком высокой, особенно на коротких сообщениях вроде `a?`

## Approach
Сделать bubble компактнее через 3 уровня:
- уменьшить внутренние вертикальные отступы
- уменьшить расстояние между текстом и временем
- сделать отдельную более компактную геометрию для коротких текстовых сообщений, если это понадобится после первой итерации

## Plan
### Phase 1: Зафиксировать целевую геометрию сообщения
- Определить, какой визуальный результат считаем правильным:
  - сообщение не выглядит “воздушным” снизу
  - время остается читаемым, но не отталкивает текст слишком далеко
  - короткие сообщения не превращаются в слишком высокий овал
- Взять текущий bubble как базу и выбрать более компактные значения:
  - `py-3` -> проверить `py-2.5` или `py-2`
  - `leading-6` -> проверить `leading-5`
  - `mt-2` у времени -> проверить `mt-1` или `mt-0.5`
- Output / exit criterion:
  - есть конкретные новые размеры для bubble, текста и времени

### Phase 2: Уменьшить вертикальный объем bubble
- В [conversation-view.tsx](C:\Users\User\Desktop\Project\apps\web\components\conversation-view.tsx) уменьшить внутренние вертикальные отступы у контейнера сообщения
- Сохранить горизонтальный комфорт чтения, то есть:
  - вертикаль уменьшаем
  - горизонталь почти не трогаем или трогаем минимально
- Проверить отдельно:
  - исходящие сообщения
  - входящие сообщения
- Output / exit criterion:
  - bubble становится ниже уже без изменения логики контента

### Phase 3: Сжать блок времени
- Уменьшить расстояние между основным текстом и временем
- Проверить, не лучше ли сделать время чуть менее заметным по размеру и line-height, чтобы оно занимало меньше места
- Сравнить 2 варианта:
  - оставить время отдельной строкой, но ближе к тексту
  - оставить ту же структуру, но сделать line-height у времени компактнее
- Для входящих и исходящих сохранить читаемость контраста
- Output / exit criterion:
  - нижняя часть bubble перестает выглядеть пустой

### Phase 4: Проверить короткие сообщения как отдельный кейс
- Основной визуальный дефект сейчас заметнее всего на коротких сообщениях вроде:
  - `a`
  - `ok`
  - `a?`
- После базового уменьшения отступов проверить, достаточно ли этого
- Если нет, добавить мягкую дополнительную логику:
  - компактный режим для коротких текстовых сообщений без вложений
  - или более аккуратную минимальную высоту bubble
- Важно:
  - не вводить сложную ветвистую логику без необходимости
  - сначала попробовать решить все только классами
- Output / exit criterion:
  - короткие сообщения больше не выглядят непропорционально высокими

### Phase 5: Проверить входящие сообщения с именем отправителя
- У входящих сообщений есть дополнительный верхний текст с именем отправителя
- Нужно проверить, что после уплотнения:
  - имя не прилипает к тексту слишком сильно
  - bubble не выглядит перегруженным
- Возможно, придется отдельно чуть уменьшить:
  - `mb-1` у имени
  - tracking/line-height у имени
- Output / exit criterion:
  - входящие сообщения тоже становятся компактнее, но не теряют структуру

### Phase 6: Проверить сообщения с вложениями
- Сейчас в bubble могут быть:
  - текст
  - вложение
  - только вложение
- Нужно убедиться, что уплотнение текстового bubble не ломает attachment layout
- Особенно проверить:
  - текст + attachment + время
  - attachment-only message
  - image preview + caption/body
- Возможно, понадобится:
  - уменьшать отступы только для text-only сообщений
  - а attachment-сообщения оставить чуть более свободными
- Output / exit criterion:
  - новая компактная схема не ломает сообщения с файлами

## Testing and Validation
- Проверить очень короткое исходящее сообщение:
  - bubble визуально компактный
  - время не висит слишком низко
- Проверить короткое входящее сообщение:
  - имя, текст и время выглядят собранно
- Проверить длинное сообщение:
  - переносы строк читаемы
  - bubble не стал слишком тесным
- Проверить несколько сообщений подряд:
  - ритм списка сообщений выглядит аккуратно
- Проверить сообщение с вложением:
  - нет визуального конфликта с preview/file card
- Проверить desktop и mobile width
- При необходимости прогнать существующий UI e2e, если затронутся `message-item` или структура bubble

## Risks and Dependencies
- Риск: слишком сильное уменьшение `py` сделает bubble тесным и ухудшит читаемость.
- Mitigation:
  - уменьшать вертикальные отступы постепенно, в 1 итерацию за раз.

- Риск: время начнет визуально прилипать к тексту.
- Mitigation:
  - уменьшать `mt`, но не убирать разделение полностью.

- Риск: входящие сообщения с именем отправителя станут выглядеть сжато хуже, чем исходящие.
- Mitigation:
  - отдельно проверить и при необходимости слегка скорректировать spacing для incoming variant.

- Риск: compact text styling повлияет на attachment-сообщения.
- Mitigation:
  - сначала менять общие отступы, потом при необходимости вводить частичную дифференциацию для text-only и attachment messages.
